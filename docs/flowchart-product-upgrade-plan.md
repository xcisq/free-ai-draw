# Drawnix 流程图能力全面优化方案

> 版本：v1.1  
> 日期：2026-05-11  
> 状态：待评审  
> 适用范围：`packages/drawnix/src/**`、`packages/react-board/**`、流程图模板、流程图资产管理、流程图编辑体验

## 1. 文档定位

本文档用于规划 Drawnix 的通用流程图能力升级。目标不是只补几个形状或快捷按钮，而是把流程图能力从“画布上的单次绘制”升级为接近幕布、XMind 等产品的结构化生产工具。

后续涉及以下需求时，默认先对齐本文档：

- 流程图模板库
- 流程图管理与复用
- 大纲化流程编辑
- 节点层级、分支、折叠、展开
- 流程图样式主题
- 流程图导入、导出、复制、保存
- AI 生成流程图后的编辑与管理

## 2. 总目标

Drawnix 需要具备一套完整的流程图工作台能力：

1. 用户可以从模板快速创建流程图。
2. 用户可以像幕布一样用大纲维护流程结构。
3. 用户可以像 XMind 一样管理主题、分支、层级和布局。
4. 流程图可以被保存、归档、复用、复制和再次编辑。
5. 流程图既能作为画布元素存在，也能作为独立文档资产存在。
6. AI 生成的流程图不只是一次性插入画布，而是能进入后续管理、编辑和模板化复用。

最终体验应接近：

```text
选择模板 / 输入大纲 / AI 生成
        ↓
进入流程图编辑工作台
        ↓
大纲与画布双向同步
        ↓
切换布局、主题、模板
        ↓
保存为流程图资产 / 插入画布 / 导出文件
```

## 3. 产品原则

### 3.1 核心原则

- **结构优先**：流程图首先是结构化内容，其次才是图形摆放。
- **大纲可编辑**：复杂流程必须能通过文本大纲快速维护，不强迫用户只在画布上拖拽。
- **模板可复用**：模板不仅包含视觉样式，还应包含结构骨架、推荐节点、布局规则和适用场景。
- **管理可沉淀**：用户画过的流程图应能沉淀为资产，而不是散落在无限画布中。
- **画布不割裂**：流程图工作台与 Drawnix 画布要能互相转换，不能变成两个完全无关的系统。
- **AI 不越权**：AI 可以生成初稿、补全节点、推荐模板，但最终结构应可由用户明确确认和编辑。

### 3.2 当前非目标

第一阶段不优先做：

- 多人实时协作
- 企业级权限与审批
- 云端同步和跨设备文档库
- 完整知识库系统
- 在线模板市场
- 复杂项目管理甘特图

当前先聚焦本地可用、可编辑、可保存、可复用的流程图工作台。

## 4. 对标能力拆解

### 4.1 幕布类能力

幕布的核心价值不是图形，而是“用大纲组织复杂内容”。Drawnix 应吸收以下能力：

- 大纲节点快速新增、删除、缩进、提升、移动。
- 大纲层级自动映射为流程图层级。
- 支持折叠和展开某个分支。
- 支持从 Markdown / 纯文本大纲生成流程结构。
- 支持通过快捷键连续输入节点。
- 支持节点备注、标签、状态等轻量元信息。

建议 MVP 中先支持：

- Tab / Shift+Tab 调整层级。
- Enter 新增同级节点。
- Backspace 删除空节点。
- 拖拽大纲节点调整顺序。
- 大纲变更实时刷新画布结构。

### 4.2 XMind 类能力

XMind 的核心价值是“结构模板 + 主题样式 + 分支布局”。Drawnix 应吸收以下能力：

- 中心主题、主分支、子分支的层级组织。
- 多种结构布局：逻辑图、树图、时间轴、鱼骨图、矩阵图、组织结构图。
- 一键切换主题与配色。
- 分支折叠、展开、聚焦。
- 图标、标签、优先级、进度等节点标记。
- 导出图片、PDF、PPTX、Markdown。

建议 MVP 中先支持：

- 流程图、树状图、组织结构图、时间轴四类结构模板。
- 主题切换：学术简洁、商务蓝、黑白打印、彩色模块。
- 分支折叠与聚焦查看。
- 节点标记：状态、优先级、负责人、备注。

## 5. 产品形态

### 5.1 新入口：流程图工作台

建议在 Drawnix 中新增独立入口：

```text
新建 -> 流程图
```

进入后不是直接落一个空画布，而是先进入流程图工作台：

```text
+-------------------------------------------------------------+
| topbar: 返回画布 / 保存 / 插入画布 / 导出 / 模板库          |
+----------------------+--------------------------------------+
| 左侧：大纲编辑器      | 右侧：流程图预览与编辑画布           |
| - 节点层级            | - 自动布局                          |
| - 标签/备注           | - 拖拽微调                          |
| - 折叠/展开           | - 连线编辑                          |
| - 模板结构提示        | - 主题预览                          |
+----------------------+--------------------------------------+
| 底部：布局、主题、检查、AI 补全、历史版本                  |
+-------------------------------------------------------------+
```

### 5.2 双模式编辑

流程图工作台需要同时支持两种编辑方式：

| 模式 | 适用场景 | 关键能力 |
|------|----------|----------|
| 大纲模式 | 快速组织结构、批量增删节点 | 键盘输入、层级调整、折叠展开 |
| 画布模式 | 精细排版、连线调整、视觉打磨 | 拖拽节点、编辑连线、调整样式 |

两个模式必须双向同步：

- 大纲新增节点，画布立即生成节点。
- 大纲调整层级，画布重新计算结构。
- 画布拖拽节点，不应破坏大纲层级。
- 画布新增连线，需要同步为结构关系或额外关系。

## 6. 模板体系

### 6.1 模板分层

模板不能只是一张静态示例图，应拆成四层：

```text
Template = Structure + Layout + Theme + Content Preset
```

| 层级 | 说明 | 示例 |
|------|------|------|
| Structure | 结构骨架 | 线性流程、分支流程、闭环流程、组织结构 |
| Layout | 排版规则 | 从左到右、从上到下、中心发散、泳道 |
| Theme | 视觉主题 | 学术简洁、商务蓝、彩色模块、黑白打印 |
| Content Preset | 内容预设 | 项目复盘、产品路线图、审批流程、研发流程 |

这样同一个结构可以换主题，同一个主题也可以套不同结构。

### 6.2 首批模板分类

MVP 建议提供以下模板：

| 分类 | 模板 | 用途 |
|------|------|------|
| 基础流程 | 线性流程 | 标准步骤说明 |
| 基础流程 | 分支判断流程 | 带条件判断的业务流程 |
| 基础流程 | 闭环反馈流程 | 迭代、复盘、优化链路 |
| 管理协作 | 项目流程 | 需求、设计、开发、测试、上线 |
| 管理协作 | 审批流程 | 发起、审核、复核、归档 |
| 管理协作 | 责任分工 | 角色、职责、交付物 |
| 知识整理 | 树状结构 | 概念拆解、知识梳理 |
| 知识整理 | 时间轴 | 路线图、发展过程 |
| 组织结构 | 组织架构图 | 团队、部门、岗位 |
| 技术研发 | 系统流程 | 输入、处理、输出、监控 |
| 技术研发 | 数据流图 | 采集、清洗、处理、消费 |
| 技术研发 | AI 工作流 | 输入、模型、工具、输出、评估 |

### 6.3 模板元数据

模板需要具备可检索、可推荐的元数据：

```typescript
interface FlowchartTemplate {
  id: string;
  name: string;
  description: string;
  category: 'basic' | 'management' | 'knowledge' | 'org' | 'technical';
  structureType:
    | 'linear'
    | 'branch'
    | 'loop'
    | 'tree'
    | 'timeline'
    | 'org'
    | 'swimlane'
    | 'dataflow';
  layoutType: 'LR' | 'TB' | 'RADIAL' | 'TREE' | 'TIMELINE' | 'SWIMLANE';
  themeId: string;
  tags: string[];
  outlinePreset: FlowchartOutlineNode[];
  stylePreset: FlowchartStylePreset;
  previewThumbnail: string;
}
```

## 7. 流程图管理

### 7.1 管理对象

流程图需要从“画布上的一组元素”升级为可管理对象：

```typescript
interface FlowchartDocument {
  id: string;
  title: string;
  description?: string;
  createdAt: number;
  updatedAt: number;
  templateId?: string;
  themeId?: string;
  outline: FlowchartOutlineNode[];
  elements: PlaitElement[];
  thumbnail?: string;
  tags: string[];
  source: 'blank' | 'template' | 'outline' | 'ai' | 'import';
}
```

### 7.2 管理能力

流程图管理至少应支持：

- 新建流程图
- 从模板新建
- 从大纲新建
- 从 AI 生成新建
- 保存当前流程图
- 另存为模板
- 复制流程图
- 重命名流程图
- 删除流程图
- 标签筛选
- 最近编辑
- 搜索标题和节点内容
- 插入当前画布
- 从画布元素提取为流程图资产

### 7.3 本地存储策略

MVP 阶段建议优先本地存储，不引入服务端复杂度：

```text
localStorage / IndexedDB
  |- flowchart-documents
  |- flowchart-templates
  |- flowchart-thumbnails
  |- recent-flowcharts
```

后续如果进入云端版本，再把 `FlowchartDocument` 作为稳定协议迁移到后端。

## 8. 大纲数据模型

### 8.1 节点模型

```typescript
interface FlowchartOutlineNode {
  id: string;
  title: string;
  children: FlowchartOutlineNode[];
  collapsed?: boolean;
  note?: string;
  markers?: FlowchartMarker[];
  relation?: {
    type: 'next' | 'condition' | 'parallel' | 'loop' | 'reference';
    label?: string;
  };
}
```

### 8.2 标记模型

```typescript
interface FlowchartMarker {
  type: 'status' | 'priority' | 'owner' | 'tag' | 'progress';
  value: string;
}
```

示例：

```text
- 产品需求
  - 需求收集 [status: todo]
  - 需求评审 [owner: PM]
  - 方案确认 [priority: high]
- 研发实现
  - 前端开发
  - 后端开发
  - 联调测试
- 发布复盘
  - 灰度发布
  - 数据观察
  - 复盘优化
```

## 9. 布局与结构能力

### 9.1 布局类型

首批支持以下布局：

| 布局 | 说明 | 优先级 |
|------|------|--------|
| LR 流程 | 从左到右推进 | P0 |
| TB 流程 | 从上到下推进 | P0 |
| Tree 树状 | 父子层级展开 | P0 |
| Timeline 时间轴 | 按阶段顺序排列 | P1 |
| Swimlane 泳道 | 按角色或部门分区 | P1 |
| Radial 发散 | 中心主题向外展开 | P2 |
| Matrix 矩阵 | 二维维度组织 | P2 |

### 9.2 结构规则

布局不能只根据节点数量随意排布，应消费明确结构：

- 一级节点决定主干或主分支。
- 二级节点决定模块内部结构。
- `condition` 关系生成判断分支。
- `parallel` 关系生成并行分支。
- `loop` 关系生成反馈线。
- `collapsed` 节点不展开子节点，但保留聚合标识。

### 9.3 与现有 Plait 元素的关系

流程图文档需要能转换为标准 Plait 元素：

```text
FlowchartDocument
  -> FlowchartLayoutModel
  -> PlaitElement[]
  -> board.insertFragment
```

反向也需要支持：

```text
选中一组 PlaitElement
  -> 识别节点和连线
  -> 重建 FlowchartDocument
  -> 保存为流程图资产
```

## 10. 主题与样式

### 10.1 首批主题

| 主题 | 风格 | 适用场景 |
|------|------|----------|
| `academic-clean` | 白底、细线、低饱和 | 文档、论文、报告 |
| `business-blue` | 蓝灰配色、稳重 | 商务汇报 |
| `color-modules` | 模块分色 | 多阶段、多部门流程 |

| `soft-card` | 卡片式、轻阴影 | 产品说明、知识整理 |

### 10.2 样式规则

主题应覆盖：

- 节点填充色
- 节点边框色
- 节点圆角
- 文本字体、字号、颜色
- 连线颜色、粗细、箭头
- 模块背景色
- 折叠节点样式
- 标记图标样式

主题切换必须保留结构，不应改变节点文本、层级和关系。

## 11. UI 视觉与交互设计方案

### 11.1 设计方向

流程图工作台采用“学术编辑台 / paper workbench”方向，而不是默认 AI 工具常见的紫蓝渐变风格。整体应像一个专业的结构化编辑桌面：干净、安静、信息密度高，但关键操作清晰可见。

- **Visual thesis**：浅色纸面工作台，深墨文字、克制边框、编辑器级密度，用少量蓝绿色状态色提示当前结构状态。
- **Content plan**：先让用户知道当前位置，再展示大纲与画布双栏，最后提供模板、主题、保存、插入画布等明确操作。
- **Interaction thesis**：按钮按压轻微缩放，模板卡片悬浮时露出结构预览，画布节点变更时使用短促的局部过渡而不是整屏动画。

### 11.2 UI 总体骨架

流程图工作台推荐使用三栏式专业编辑器结构：

```text
+--------------------------------------------------------------------------------+
| Topbar                                                                         |
| 返回画布 / 面包屑 / 文档名 / 保存状态 / 模板库 / 导出 / 插入画布                |
+----------------------+----------------------------------------+----------------+
| Outline Panel        | Canvas Preview                         | Inspector      |
| 大纲、层级、标记      | 流程图实时预览、拖拽、缩放、连线编辑       | 模板、主题、属性 |
| 快捷输入、折叠展开    | 当前布局、结构警告、聚焦模式              | 节点设置、关系设置 |
+----------------------+----------------------------------------+----------------+
| Bottom Status Bar                                                               |
| 节点数 / 连线数 / 布局类型 / 未保存提示 / 结构检查结果                          |
+--------------------------------------------------------------------------------+
```

默认尺寸建议：

| 区域 | 桌面端宽度 | 说明 |
|------|------------|------|
| Topbar | `100% x 52px` | 承载导航、状态和主操作 |
| Outline Panel | `300-360px` | 适合大纲编辑和键盘操作 |
| Canvas Preview | 自适应 | 保证流程图预览是视觉中心 |
| Inspector | `300-340px` | 模板、主题、属性设置 |
| Bottom Status Bar | `100% x 34px` | 展示轻量诊断和状态 |

### 11.3 页面视觉规范

### 11.3.1 色彩系统

基础色彩应延续 Drawnix 当前工作台的浅色纸面风格：

| Token | 建议值 | 用途 |
|------|--------|------|
| `--flow-bg` | `#f7f8fb` | 页面主背景 |
| `--flow-surface` | `#ffffff` | 面板、卡片、弹窗 |
| `--flow-surface-soft` | `#f9fafb` | 次级面板背景 |
| `--flow-border` | `#e4e7ec` | 默认边框 |
| `--flow-border-strong` | `#d0d5dd` | 强边框、选中边框 |
| `--flow-text` | `#111827` | 主文字 |
| `--flow-text-secondary` | `#344054` | 次级文字 |
| `--flow-text-muted` | `#667085` | 辅助说明 |
| `--flow-accent` | `#2563eb` | 当前选中、主操作 |
| `--flow-success` | `#16a34a` | 保存成功、检查通过 |
| `--flow-warning` | `#d97706` | 结构风险、未保存 |
| `--flow-danger` | `#dc2626` | 删除、错误 |

颜色使用规则：

- 主背景保持浅灰纸面，不使用大面积渐变。
- 只有主操作、选中态和状态反馈可以使用高饱和颜色。
- 模板缩略图可以更丰富，但不能污染主工作台界面。
- 危险色只用于删除、错误和不可恢复操作。

### 11.3.2 字体与层级

字体体系：

| 内容 | 字体 | 字号 | 字重 |
|------|------|------|------|
| 页面标题 | `Songti SC / STSong / Noto Serif CJK SC` | `18-22px` | `600` |
| 面板标题 | `PingFang SC / Helvetica Neue / Arial` | `13-14px` | `600` |
| 正文与控件 | `PingFang SC / Helvetica Neue / Arial` | `13px` | `400-500` |
| 大纲节点 | `PingFang SC / Helvetica Neue / Arial` | `13-14px` | `400-600` |
| 辅助说明 | `Songti SC / STSong / Times New Roman` | `12px` | `400` |
| 状态数字 | `SFMono-Regular / Menlo / Consolas` | `11-12px` | `500` |

层级规则：

- 页面标题不做大号营销化标题，应像文档编辑器标题。
- 大纲节点需要比普通说明更清晰，选中节点使用浅蓝底和左侧竖线。
- 辅助说明可以使用衬线斜体，但必须克制。
- 状态栏数字、节点数、连线数使用等宽字体，增强工具感。

### 11.3.3 间距与圆角

间距建议：

| Token | 值 | 用途 |
|------|----|------|
| `--space-1` | `4px` | 图标间距、紧凑控件 |
| `--space-2` | `8px` | 按钮内边距、列表间距 |
| `--space-3` | `12px` | 卡片内边距 |
| `--space-4` | `16px` | 面板内边距 |
| `--space-6` | `24px` | 大区块间距 |

圆角建议：

| Token | 值 | 用途 |
|------|----|------|
| `--radius-sm` | `6px` | 小按钮、标签 |
| `--radius-md` | `10px` | 输入框、列表项 |
| `--radius-lg` | `14px` | 卡片、面板 |
| `--radius-pill` | `999px` | 状态 chip、筛选项 |

规则：

- 同一类组件只使用一个圆角值，不混用多套半径。
- 面板圆角大于控件圆角，卡片圆角介于两者之间。
- 不使用厚重投影模拟“高级感”，优先靠背景层级和细边框建立空间。

### 11.4 核心页面设计

### 11.4.1 流程图工作台首页

工作台首页用于“新建或打开流程图”，应避免空白画布造成无从下手。

```text
+-------------------------------------------------------------+
| Flowchart Workbench                                         |
| 结构化流程图，把大纲、模板和画布放在同一个编辑台里            |
+-------------------------------------------------------------+
| [从空白开始] [从模板开始] [粘贴大纲生成] [AI 生成初稿]        |
+--------------------------+----------------------------------+
| 最近编辑                  | 推荐模板                          |
| - 项目上线流程             | [项目流程] [审批流程] [AI 工作流]   |
| - 数据处理链路             | [组织结构] [时间轴] [树状拆解]      |
+--------------------------+----------------------------------+
```

设计要点：

- 页面不做大 hero，不使用营销文案。
- 主操作按钮不超过 4 个，默认推荐“从模板开始”。
- 最近编辑以紧凑列表展示，显示标题、更新时间、节点数。
- 推荐模板以小型卡片展示结构缩略图，悬浮时显示“使用此模板”。

### 11.4.2 大纲编辑面板

大纲面板是产品的核心，应做成轻量但高效的编辑器。

```text
+--------------------------------+
| 大纲                            |
| 搜索节点...                     |
+--------------------------------+
| 1  产品需求                     |
|    1.1  需求收集      [todo]    |
|    1.2  需求评审      [PM]      |
|    1.3  方案确认      [high]    |
| 2  研发实现                     |
|    2.1  前端开发                |
|    2.2  后端开发                |
|    2.3  联调测试                |
+--------------------------------+
| Enter 新增  Tab 降级  Shift+Tab 提级 |
+--------------------------------+
```

设计规则：

- 每个大纲节点高度建议 `30-34px`。
- 当前选中节点使用 `#eff6ff` 浅蓝背景和 `2px` 左侧强调线。
- 折叠节点显示子节点数量，例如 `研发实现 · 3 项`。
- 节点拖拽时显示细线落点，不使用厚重阴影。
- 节点标记使用小号 chip，避免抢占标题。

交互规则：

- `Enter` 新增同级节点。
- `Tab` 降为子级。
- `Shift+Tab` 提升一级。
- `Cmd/Ctrl + ArrowUp/ArrowDown` 调整节点顺序。
- `Cmd/Ctrl + /` 打开快捷键说明。
- 编辑时只更新当前节点对应的画布区域，避免整图跳动。

### 11.4.3 画布预览区

画布预览区是结构结果的可视化中心，不应被工具栏挤压。

```text
+-------------------------------------------------------------+
| [LR] [TB] [Tree] [Timeline]     结构检查：通过               |
+-------------------------------------------------------------+
|                                                             |
|    +----------+      +----------+      +----------+          |
|    | 产品需求 | ---> | 研发实现 | ---> | 发布复盘 |          |
|    +----------+      +----------+      +----------+          |
|          |                 |                 |               |
|       子节点             子节点             子节点            |
|                                                             |
+-------------------------------------------------------------+
```

设计规则：

- 预览区背景使用低对比网格纸纹理，网格线透明度要低。
- 当前选中节点和大纲选中状态联动。
- 结构检查状态放在右上角，不要做成弹窗打断。
- 缩放控件放在右下角，包含适配画布、缩小、放大。
- 布局切换按钮放在预览区顶部，使用 segmented control。

动效规则：

- 节点新增：`120ms` fade + slight scale。
- 节点删除：`90ms` fade out。
- 布局切换：`180-220ms` position transition。
- 连线重算：只过渡路径，不闪烁节点。

### 11.4.4 模板库界面

模板库需要像专业模板浏览器，而不是简单图标网格。

```text
+-------------------------------------------------------------+
| 模板库                                      搜索模板...       |
+--------------------+----------------------------------------+
| 分类                | 模板详情                               |
| 全部                | +----------------+ +----------------+   |
| 基础流程            | | 项目流程        | | 审批流程        |   |
| 管理协作            | | preview graph   | | preview graph   |   |
| 知识整理            | | 12 节点 · LR    | | 8 节点 · Branch |   |
| 技术研发            | +----------------+ +----------------+   |
+--------------------+----------------------------------------+
```

模板卡片设计：

- 卡片比例建议 `4:3` 或 `16:10`。
- 上半部分显示真实结构缩略图，不使用抽象插画代替。
- 下半部分显示模板名、适用场景、节点数量、布局类型。
- 鼠标悬浮时显示两项操作：“预览”和“使用”。
- 推荐模板可加 `推荐` chip，但不使用大面积彩色标签。

模板详情页：

- 左侧展示大预览。
- 右侧展示适用场景、默认大纲、可切换主题。
- 底部提供“使用模板”和“复制为自定义模板”。

### 11.4.5 流程图管理面板

流程图管理面板应接近轻量文档库，支持快速查找和复用。

```text
+-------------------------------------------------------------+
| 流程图管理                              [新建] [导入]        |
+-------------------------------------------------------------+
| 搜索标题、节点内容、标签...                                  |
+-------------------------------------------------------------+
| 最近编辑                                                     |
| [缩略图] 项目上线流程      32 节点 · 5 条分支 · 2 分钟前       |
| [缩略图] 数据处理链路      18 节点 · Tree · 昨天              |
| [缩略图] AI Agent 工作流   24 节点 · Swimlane · 3 天前        |
+-------------------------------------------------------------+
```

设计规则：

- 列表默认比网格更高效，尤其适合文档管理。
- 每项必须展示缩略图、标题、更新时间、节点数、模板来源。
- 右侧 hover 操作包含打开、插入画布、复制、删除。
- 删除需要二次确认，避免误删本地资产。

### 11.4.6 右侧属性检查器

Inspector 不应堆满所有功能，推荐分 Tab：

```text
[属性] [主题] [模板] [检查]
```

各 Tab 内容：

| Tab | 内容 |
|-----|------|
| 属性 | 当前节点标题、备注、标记、关系类型 |
| 主题 | 当前主题、颜色、节点样式、连线样式 |
| 模板 | 当前模板信息、替换模板、另存为模板 |
| 检查 | 孤立节点、断裂连线、循环风险、未保存状态 |

设计规则：

- 每个 Tab 只做一件事，避免滚动很长的设置面板。
- 属性变更需要即时预览，但保存状态要明确提示。
- 结构检查结果用可展开列表，不用全局 toast 刷屏。

### 11.5 组件设计规范

### 11.5.1 按钮

按钮分为四类：

| 类型 | 用途 | 样式 |
|------|------|------|
| Primary | 插入画布、使用模板、保存 | 深墨或蓝色底，白字 |
| Secondary | 预览、复制、导出 | 白底，细边框 |
| Ghost | 面板内轻量操作 | 透明底，hover 浅灰 |
| Danger | 删除 | 红色文字或红色浅底 |

交互：

- 所有按钮 `:active` 使用 `transform: scale(0.96)`。
- 主按钮 hover 不改变尺寸，只改变背景深浅。
- loading 状态保留原按钮宽度，避免布局抖动。

### 11.5.2 输入框

输入框设计：

- 背景 `#ffffff`。
- 边框 `#d0d5dd`。
- 高度 `32-36px`。
- focus 使用深墨边框或轻量 focus ring。
- placeholder 使用 `#98a2b3`。

大纲编辑器中的输入态不使用传统输入框边框，而是直接在列表项内编辑，避免破坏大纲连续性。

### 11.5.3 Chip 与标签

标签用于状态、角色、优先级和模板分类：

| 类型 | 示例 | 颜色 |
|------|------|------|
| 状态 | `todo` / `doing` / `done` | 灰、蓝、绿 |
| 优先级 | `high` / `low` | 橙、灰 |
| 角色 | `PM` / `FE` / `BE` | 蓝灰 |
| 模板 | `Tree` / `LR` / `Swimlane` | 中性灰 |

标签高度建议 `20-22px`，字体 `11-12px`，圆角 pill。

### 11.6 响应式设计

桌面端优先三栏：

```text
Outline 320px + Canvas auto + Inspector 320px
```

中等宽度降为两栏：

```text
Outline 300px + Canvas auto
Inspector 变为右侧抽屉
```

窄屏降为单栏：

```text
Topbar
Segmented tabs: 大纲 / 预览 / 属性
当前 tab 内容
Bottom action bar
```

窄屏规则：

- 保留返回、保存、插入画布三个关键操作。
- 模板库默认全屏弹出，不塞进侧栏。
- 大纲编辑优先于画布微调。
- 画布预览允许只读缩放，复杂拖拽可降级。

### 11.7 可访问性与键盘操作

必须支持：

- 所有按钮和列表项可键盘聚焦。
- 焦点态可见，不能只靠颜色区分。
- 大纲节点支持键盘新增、删除、缩进、移动。
- 模板卡片支持 Enter 使用、Space 预览。
- 色彩对比度满足常规可读性要求。
- 结构检查信息不能只用红黄绿表达，必须有文字说明。

### 11.8 UI 验收标准

UI 设计完成后至少满足：

- 第一屏能明确看出这是“流程图工作台”，不是普通画布或聊天生成器。
- 用户 5 秒内能找到“从模板开始、编辑大纲、插入画布”三个关键操作。
- 大纲、画布、属性三块区域层级清晰，不互相抢视觉中心。
- 模板卡片能看出真实结构差异，而不是只换封面颜色。
- 主题切换后，节点、连线、模块、文本风格保持一致。
- 所有主要按钮有清晰 hover、active、disabled、loading 状态。
- 375px 宽度下不出现内容横向溢出，关键操作不丢失。

## 12. AI 辅助能力

AI 能力应围绕“结构初稿”和“编辑辅助”，而不是替代用户决策。

### 12.1 AI 入口

- 文本生成流程图
- 大纲补全
- 节点命名优化
- 自动归纳阶段
- 推荐模板
- 推荐主题
- 检查流程缺失步骤
- 将画布流程转为大纲

### 12.2 AI 生成后的确认机制

AI 输出必须进入可编辑结构层：

```text
AI 输出
  -> FlowchartOutlineNode[]
  -> 用户确认大纲
  -> 自动布局
  -> 用户编辑画布
  -> 保存为 FlowchartDocument
```

不得只生成一组散落的画布元素后结束。

## 14. 与现有能力的关系

### 14.1 与 PaperDraw 的关系

PaperDraw 面向论文级 pipeline 图，强调从论文文本生成高质量学术流程图。通用流程图工作台面向更广泛的管理、知识整理、业务流程和技术流程。

两者应复用底层能力，但产品入口和目标不同：

| 能力 | PaperDraw | 通用流程图工作台 |
|------|-----------|------------------|
| 目标 | 论文 pipeline 图 | 通用结构化流程图 |
| 输入 | 方法描述文本 | 模板、大纲、AI、画布选择 |
| 核心 | 语义解析、布局优化、论文风格 | 大纲编辑、模板管理、资产复用 |
| 输出 | 插入画布的论文流程图 | 可保存、可管理的流程图文档 |

### 14.2 与现有 Shape Picker 的关系

Shape Picker 负责低层图形选择。流程图工作台负责高层结构创建。

不建议把所有流程图能力都塞进 Shape Picker。Shape Picker 只保留：

- 基础形状
- 流程图符号
- UML / Swimlane / Table 等图形工具

流程图模板、管理、大纲编辑应进入独立工作台。

### 14.3 与资产库的关系

流程图文档可以作为一种新的资产类型：

```typescript
type AssetKind = 'image' | 'svg' | 'drawnix' | 'flowchart';
```

`flowchart` 资产与 `.drawnix` 资产的区别：

- `.drawnix` 保存的是画布片段。
- `flowchart` 保存的是可重新生成布局的大纲、模板、主题和元素。

## 15. 分阶段执行计划

## 15.1 阶段 F0：需求冻结与基线梳理

### 目标

明确流程图工作台与现有 PaperDraw、Shape Picker、资产库的边界。

### 任务

- 梳理现有流程图相关入口。
- 确定新入口位置。
- 确定首批模板分类。
- 确定本地存储方式。
- 冻结 `FlowchartDocument` 与 `FlowchartOutlineNode` 基础 schema。

### 交付物

- 最终 PRD
- 数据模型草案
- 首批模板清单
- 入口交互草图

### 验收标准

- 团队能明确判断一个需求属于 PaperDraw、Shape Picker 还是流程图工作台。
- 首批模板和文档数据结构不再频繁变化。

## 15.2 阶段 F1：流程图工作台 MVP

### 目标

建立一个可用的流程图工作台骨架。

### 任务

- 新增流程图入口。
- 新增工作台弹窗或独立页面。
- 实现左侧大纲编辑器。
- 实现右侧流程图预览。
- 支持 `LR / TB / Tree` 三种布局。
- 支持插入主画布。

### 交付物

- `flowchart-workbench`
- `flowchart-outline-panel`
- `outline-to-layout`
- `flowchart-to-elements`

### 验收标准

- 用户能从空白大纲创建流程图。
- 用户能通过大纲新增、删除、调整层级。
- 画布预览能实时更新。
- 能插入 Drawnix 主画布并保持可编辑。

## 15.3 阶段 F2：模板库

### 目标

让用户可以从模板快速创建流程图。

### 任务

- 新增模板库面板。
- 内置首批 10-12 个模板。
- 每个模板提供预览图、说明、标签。
- 支持模板搜索和分类筛选。
- 支持从模板创建流程图文档。

### 交付物

- `flowchart-template-gallery`
- `builtin-templates`
- `template-registry`

### 验收标准

- 用户能按分类浏览模板。
- 用户能从模板一键生成可编辑大纲和图。
- 模板切换主题后结构不丢失。

## 15.4 阶段 F3：流程图管理

### 目标

让流程图可以保存、查找、复用。

### 任务

- 实现本地流程图文档存储。
- 新增流程图管理面板。
- 支持最近编辑、搜索、标签、复制、删除。
- 支持从当前画布选区保存为流程图。
- 支持流程图重新打开并继续编辑。

### 交付物

- `flowchart-storage`
- `flowchart-manager`
- `elements-to-flowchart`

### 验收标准

- 用户能保存多个流程图。
- 用户能重新打开历史流程图继续编辑。
- 用户能从画布选区生成流程图资产。
- 保存后的流程图能再次插入画布。

## 15.5 阶段 F4：主题与样式系统

### 目标

让流程图具备稳定的视觉主题和一键换肤能力。

### 任务

- 新增主题面板。
- 实现首批 5 个主题。
- 支持节点、连线、模块、标记统一换肤。
- 支持用户微调局部样式。
- 支持保存为自定义主题。

### 交付物

- `flowchart-theme-panel`
- `flowchart-style-preset`
- 主题测试样例

### 验收标准

- 切换主题不改变结构。
- 主题应用后节点、连线、文本、模块样式一致。
- 黑白打印主题在导出图片时仍可读。

## 15.6 阶段 F5：AI 与高级结构

### 目标

把 AI 生成和高级图结构接入流程图工作台。

### 任务

- 文本转大纲。
- AI 推荐模板。
- AI 补全缺失步骤。
- 支持 Timeline / Swimlane / Radial / Matrix 布局。
- 支持分支折叠、聚焦查看。
- 支持节点备注、状态、优先级、负责人。

### 交付物

- `flowchart-ai-pipeline`
- 高级布局引擎
- 标记与备注 UI

### 验收标准

- AI 生成结果进入可编辑大纲，而不是直接生成不可控画布元素。
- 用户能从 AI 初稿继续手动编辑。
- 高级布局能覆盖真实业务流程、项目流程和知识整理场景。

## 16. 验收标准

第一阶段完整交付后，应满足：

- 用户能从空白、模板、大纲三种方式创建流程图。
- 流程图大纲和画布预览双向同步。
- 至少支持 `LR / TB / Tree` 三种布局。
- 至少内置 10 个可用模板。
- 至少内置 5 个主题。
- 流程图可以保存、搜索、复制、删除、重新打开。
- 流程图可以插入主画布并保持节点与连线可编辑。
- 可以从主画布选区保存为流程图资产。
- AI 生成的流程图进入同一套文档和管理体系。

## 17. 风险与注意事项

### 17.1 不要把模板做成静态图片

模板必须是结构化数据，能够继续编辑、换主题、换布局。

### 17.2 不要让大纲和画布割裂

如果大纲无法解释画布上的结构，后续管理、模板化和 AI 编辑都会变得困难。

### 17.3 不要过早引入云端复杂度

当前应先把本地文档模型、模板体系和编辑体验做稳，再考虑云端同步。

### 17.4 不要和 PaperDraw 混成一个入口

PaperDraw 是论文 pipeline 专项能力，流程图工作台是通用结构化编辑能力。两者可以共享底层布局和元素构建，但入口和用户心智应保持清晰。
