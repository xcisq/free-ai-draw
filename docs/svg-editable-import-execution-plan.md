## 2. 背景与问题定义

## 2.1 当前能力

目前 Drawnix 已具备以下能力：

- 支持将图片文件插入为画板中的图片元素。
- 图片类型支持 `image/svg+xml`，也即 SVG 可以作为图片被插入。
- 插入后的图片元素可参与现有画板对象交互。
- 支持通过 Mermaid 等外部表示转换为 `PlaitElement[]` 并插入画板。

当前缺口是：

- 不支持将 SVG 内容结构化拆解为 Drawnix 可编辑对象。
- “打开文件”不支持将 SVG 当作 Drawnix 原生结构导入。
- 导入 SVG 后无法直接编辑其中的文字、箭头或结构关系。

## 2.2 真实需求

本次需求并不是“任意 SVG 完整无损变成通用矢量编辑对象”，而是更具体的能力：

1. SVG 中的文字需要尽可能转成可编辑文本。
2. SVG 中表达连接关系的箭头仅在连接性明确时转成可编辑连线。
3. 箭头如果本身是图片，或更适合保留原 SVG 形态，则不强制转换，允许直接作为组件导入。
4. 降级后的图片仍然必须是画板中的正常对象，能够移动、缩放、复制、删除。
5. 组件图片默认应保留透明背景，不能因为导入过程强制铺白底。

## 2.3 核心判断

SVG 是视觉格式，不是语义格式。

这意味着：

- 文本节点通常可直接恢复语义。
- 一部分连接线可以从几何和样式上识别为箭头。
- 大量复杂 `path`、装饰图形、截图型内容、嵌入图片，本质上并不适合首版做完整可编辑。

因此，本方案采用 **半结构化导入**：

- 可识别部分：转 Drawnix 可编辑元素。
- 不可识别、不值得结构化，或保留原 SVG 形态更合适的部分：转透明组件图片。

## 3. 产品目标与非目标

## 3.1 目标

本轮目标是建立一条稳定、可迭代的 SVG 导入链路：

1. 允许用户把结构化 SVG 导入为“可编辑文本 + 可编辑连接箭头 + 透明组件图”的混合结果。
2. 尽量保留原图阅读顺序、布局关系和视觉完整性。
3. 不支持的复杂结构不应导致导入失败，而应自动降级。
4. 降级后的组件仍可作为画板对象参与编辑。
5. 每轮改动都能通过样例 SVG、单测和验收标准验证。

## 3.2 非目标

当前阶段不优先处理：

- 任意 SVG 的全量无损矢量编辑
- 通用 `path` 节点级编辑器
- 渐变、滤镜、遮罩、裁剪路径的完整结构化映射
- 任意复杂贝塞尔曲线的精确可编辑还原
- 富文本、外链字体、排版引擎级文字还原
- 完整替代 Figma / Illustrator / Inkscape 之类的通用矢量编辑能力

## 4. 方案总述

## 4.1 导入结果模型

导入结果由三类对象组成：

### A. 可编辑文本

来源：

- `<text>`
- 可合并的 `<tspan>`

落板结果：

- Drawnix / Plait 文本元素

保留属性：

- 文本内容
- 坐标
- 字号
- 字体族
- 填充颜色
- 对齐方式

### B. 可编辑连接箭头

来源：

- `line`
- `polyline`
- 一部分开口 `path`
- 具有明显连接特征的线段组合

落板结果：

- Drawnix 箭头线 / 连线元素

适用前提：

- 该箭头主要承担连接关系
- 其几何意图清晰，转为连线后不会明显损失语义

保留属性：

- 起终点
- 折线路径
- 描边颜色
- 描边宽度
- 箭头方向

### C. 透明组件图片

来源：

- 复杂图标
- 装饰性箭头
- 本身就是图片的箭头
- 更适合保留原 SVG 形态的箭头
- 无法稳定识别的复杂路径组合
- SVG 中内嵌的 `<image>`
- 不适合拆散成单个可编辑图元的大块局部视觉组件

落板结果：

- Drawnix 图片元素

要求：

- 保持透明背景
- 保持原始相对位置
- 可在画板中正常调整

## 4.2 核心设计原则

- 先保证“能稳定导入”，再提高结构化比例。
- 先优先文本和连接关系，再考虑基础图形。
- 优先保持阅读结构，不追求像素级还原。
- 优先使用 SVG 片段作为组件图片，只有必要时才退化为 PNG。
- 不让单个复杂节点拖垮整张图的导入成功率。

## 5. 总体架构

目标链路如下：

1. 原始 `SVG string`
2. `ParsedSvgTree`
3. `NormalizedSvgGraph`
4. `ImportBlueprint`
5. `PlaitElement[]`
6. 插入画板

## 5.1 分层职责

### 5.1.1 解析层 `parser`

职责：

- 使用 `DOMParser` 解析 SVG
- 提取根节点、样式、元素树
- 读取基础属性、类名、行内样式
- 收集 `defs`、`style` 等共享定义

输出：

- `ParsedSvgTree`

### 5.1.2 归一化层 `normalizer`

职责：

- 展开样式类
- 归一化 `fill/stroke/stroke-width/font-size/text-anchor`
- 归一化坐标系
- 处理 `viewBox`
- 处理 `transform` 的常见情况，如 `translate/scale/rotate`
- 形成统一的图元表达

输出：

- `NormalizedSvgNode[]`

### 5.1.3 分类层 `classifier`

职责：

- 识别文本节点
- 识别候选箭头 / 连线
- 识别复杂组件边界
- 判定哪些元素转可编辑，哪些元素需要降级

输出：

- `ImportBlueprint`

### 5.1.4 组件切片层 `fragmenter`

职责：

- 将复杂组件导出为最小 SVG 片段
- 计算局部包围盒
- 排除无关背景
- 生成透明组件图的 URL / Blob / Data URL

输出：

- `ComponentAsset[]`

### 5.1.5 映射层 `mapper`

职责：

- 文本 -> 文本元素
- 连线 -> 箭头元素
- 组件图 -> 图片元素
- 保持空间位置与层级顺序

输出：

- `PlaitElement[]`

### 5.1.6 插板层 `inserter`

职责：

- 将导入结果插入当前画板
- 保持中心对齐或原始坐标
- 支持预览后确认插入

输出：

- 画板上的实际导入结果

## 6. 数据结构建议

以下不是最终锁定接口，而是建议的中间模型。

```ts
interface NormalizedStyle {
  fill?: string;
  stroke?: string;
  strokeWidth?: number;
  opacity?: number;
  fontFamily?: string;
  fontSize?: number;
  textAnchor?: 'start' | 'middle' | 'end';
}

interface NormalizedTransform {
  translateX: number;
  translateY: number;
  scaleX: number;
  scaleY: number;
  rotate?: number;
}

type NormalizedSvgNode =
  | NormalizedTextNode
  | NormalizedPathNode
  | NormalizedRectNode
  | NormalizedPolygonNode
  | NormalizedPolylineNode
  | NormalizedLineNode
  | NormalizedCircleNode
  | NormalizedEllipseNode
  | NormalizedImageNode
  | NormalizedGroupNode;

interface TextImportItem {
  kind: 'text';
  id: string;
  text: string;
  x: number;
  y: number;
  style: NormalizedStyle;
}

interface ArrowImportItem {
  kind: 'arrow';
  id: string;
  points: [number, number][];
  style: NormalizedStyle;
  arrowStart?: boolean;
  arrowEnd?: boolean;
}

interface ComponentImportItem {
  kind: 'component';
  id: string;
  bbox: { x: number; y: number; width: number; height: number };
  sourceNodeIds: string[];
  preferSvgAsset: boolean;
}

interface ImportBlueprint {
  texts: TextImportItem[];
  arrows: ArrowImportItem[];
  components: ComponentImportItem[];
  warnings: string[];
}
```

## 7. 导入规则

## 7.1 背景处理规则

### 规则 1：忽略全画布背景底板

如果某个元素满足以下特征，则判定为“全局背景”，默认不导入：

- 处于顶层或近顶层
- 是大面积 `rect`
- 宽高覆盖整张 SVG 或接近整张 SVG
- 填充为纯白或纯色
- 主要作用是画布底色，而非组件内容

目标：

- 避免导入后整张图带不必要白底
- 保持整体背景透明

### 规则 2：保留局部白底

以下情况不能误删：

- 浏览器窗口白色内容区
- 卡片背景
- 面板底色
- 图表区域底板

判断依据：

- 其几何范围是局部区域，不覆盖整张 SVG
- 它隶属于某个组件结构

## 7.2 文本导入规则

### 规则 3：独立文本优先结构化

以下文本节点应优先转可编辑：

- 独立 `<text>`
- 单行 `<text>`
- 多行但由多个独立 `<text>` 组成的标签

### 规则 4：文本样式最小保留

首版至少保留：

- 内容
- 坐标
- 字号
- 字体族
- 填充色
- 对齐方式

不强求首版完整保留：

- `dominant-baseline`
- 复杂 `tspan` 排版
- 字距、行距、字重的精确还原

### 规则 5：文本与组件可以分离

如果一个复杂组件本体降级为图片，但其标签是独立文本，则标签仍可单独转为可编辑文本。

这是本方案的核心价值之一。

## 7.3 箭头导入规则

### 规则 6：优先导入“连接性箭头”

优先识别以下箭头：

- 明显线性连接关系
- 折线或直线结构
- 路径数量少、几何意图清晰
- 更像流程关系而不是装饰图形

满足这类条件的箭头，优先转成 Drawnix 可编辑连线。

### 规则 7：箭头若保留原形态更合理，则不强制结构化

以下情况，箭头默认不强行转可编辑连线：

- 箭头本身就是 `<image>`
- 箭头位于复杂局部组件内部，拆开后会破坏视觉完整性
- 箭头是闭合填充形状，主要承担视觉表达而不是连接表达
- 箭头转成普通连线后会明显损失原图语义或外观

这类箭头应优先按“原 SVG 局部片段”导入，作为透明组件图片保留。

### 规则 8：装饰性闭合箭头默认降级

以下箭头首版默认不强行结构化：

- 闭合填充大箭头
- 曲面装饰箭头
- 复杂视觉箭头块
- 以形状为主、不是连线为主的箭头

这些箭头降级为透明组件图更稳定。

### 规则 9：箭头识别优先级

箭头识别优先顺序：

1. `line`
2. `polyline`
3. 开口 `path`
4. 带明显三角箭头特征的组合路径

## 7.4 组件导入规则

### 规则 10：剩余未识别元素按组件聚合

在提取文本和连接箭头后，剩余元素按以下方式形成组件：

- 优先按父级 `g` 分组
- 若无清晰分组，则按空间邻近和样式一致性聚合
- 避免把整张图剩余内容一次性打成一张大图

同时应额外纳入以下对象：

- 本身就是图片的箭头
- 更适合保留原 SVG 形态的箭头

### 规则 11：组件图优先导出为 SVG 片段

组件资产生成优先级：

1. 局部 SVG 片段
2. PNG fallback

只有在以下情况下才退化为 PNG：

- `foreignObject`
- 复杂 `filter`
- 浏览器对片段 SVG 渲染不稳定
- 外部兼容性要求无法满足

### 规则 12：组件图必须保留透明背景

要求：

- 不拼入全局白底
- 不在 canvas 上主动铺白色底色
- SVG 片段根节点默认透明

## 8.1 顶层背景

以下元素应忽略：

- 顶层白色背景 `svg_1`

理由：

- 覆盖全画布
- 纯白底
- 不属于具体组件内容

## 8.2 建议转为可编辑文本的部分

优先转可编辑文本的内容包括但不限于：

- 左上说明文字
- 中部标题文字
- 中部流程说明文字
- 右侧系统标题文字
- 底部标签文字

特点：

- 均为独立 `<text>`
- 坐标清晰
- 样式统一
- 无需 OCR

## 8.3 建议转为可编辑连接箭头的部分

优先转箭头的部分包括：

- `stroke-main` 的开口连接路径
- 浏览器示意图内部的连接线
- 表达明显流程关系的直线或折线箭头

暂不强求转箭头的部分：

- `fill-arrow` 闭合填充箭头
- 弯曲装饰箭头
- 本身为 `<image>` 的箭头

## 8.4 建议转为透明组件图的部分

推荐作为组件图的区域包括：

- 左上对话气泡背景
- 云朵和装饰圆点
- 中部大块视觉结构
- 右侧系统大模块容器
- 浏览器窗口整体壳层与内部复杂图示
- 需要保留原 SVG 形态的装饰箭头
- 所有嵌入 `<image>` 的内容

对这些组件，建议：

- 保留原始视觉
- 输出局部 SVG 片段
- 作为图片元素插入画板

## 9. UI 与交互建议

## 9.1 新增独立导入入口

不建议复用现有“插入图片”入口直接完成结构化导入。

建议新增入口：

- “导入 SVG 为可编辑内容”

与现有入口区分：

- 现有“插入图片”：永远作为图片导入
- 新“SVG 可编辑导入”：执行半结构化解析

## 9.2 导入后反馈

建议在导入完成后返回结构化摘要：

- 已转换文本数量
- 已转换连接箭头数量
- 已生成组件图数量
- 已忽略背景数量
- 降级原因列表

示例：

- 转换 18 个文本元素
- 转换 9 个连接箭头元素
- 生成 6 个透明组件图
- 忽略 1 个全局背景层
- 3 个复杂路径或原形态箭头已降级为组件图片

## 9.3 预览能力

建议支持导入前预览或开发态调试视图：

- 高亮哪些内容被识别为文本
- 高亮哪些内容被识别为连接箭头
- 高亮哪些区域将被切成组件图片

这有助于调试分类策略，也有助于用户理解导入结果。

## 10. 开发阶段计划

## 10.1 阶段 P0：样例基线与可观测性

### 目标

先把 SVG 导入过程看清楚，建立真实样例基线和调试输出。

### 任务

- 收集 10-20 份典型 SVG 样例
- 至少包含：
  - draw.io 导出 SVG
  - Figma 导出 SVG
  - Mermaid 导出 SVG
  - 包含内嵌 `<image>` 的复合 SVG
  - 包含背景层的流程图型 SVG
  - 本次样例 `final_edited.svg`
- 输出每份样例的元素统计：
  - 文本数
  - 路径数
  - 图像数
  - 复杂节点数
  - 背景候选数
- 建立导入调试 trace：
  - 原始节点
  - 归一化节点
  - 分类结果
  - 降级结果

### 交付物

- 样例清单
- 分类统计脚本
- 调试 trace 输出格式

### 验收标准

- 任意样例都可导出分类统计
- 团队能快速判断失败原因来自解析、分类还是组件切片

## 10.2 阶段 P1：解析与归一化

### 目标

建立一条可靠的 SVG 基础读取链路。

### 任务

- 解析 SVG DOM
- 解析 `style` 与类样式
- 支持常见属性读取
- 支持常见坐标变换
- 输出统一 `NormalizedSvgNode`

### 首版支持

- `svg`
- `g`
- `text`
- `tspan` 基础合并
- `rect`
- `line`
- `polyline`
- `polygon`
- `circle`
- `ellipse`
- `path`
- `image`

### 交付物

- `parser`
- `normalizer`
- 中间模型定义

### 验收标准

- 样例 SVG 可稳定输出归一化结果
- 样式和坐标无明显错乱

## 10.3 阶段 P2：文本与连接箭头结构化

### 目标

优先落地最有价值的可编辑能力。

### 任务

- 文本节点分类与抽取
- 文本样式最小保留
- 连接箭头识别
- 折线/直线映射为 Drawnix 连线
- 基础 warning 机制

### 交付物

- `classifier` 第一版
- 文本映射器
- 箭头映射器

### 验收标准

- 典型流程图样例中，绝大多数独立文本可编辑
- 明显连接箭头可恢复为可编辑连线

## 10.4 阶段 P3：透明组件图切片

### 目标

让复杂局部稳定保留，而不是硬转失败。

### 任务

- 剩余节点聚合为组件
- 背景过滤
- 组件包围盒计算
- 局部 SVG 片段导出
- 原形态箭头组件保留
- PNG fallback
- 组件图片导入画板

### 交付物

- `fragmenter`

## 13. 验收标准

首版 MVP 建议以以下标准验收：

1. 能导入典型流程图类 SVG。
2. 独立文本可编辑。
3. 主要连接箭头可编辑。
4. 本身为图片或更适合保留原 SVG 形态的箭头不会被强制转换，而是作为组件稳定导入。
5. 复杂局部不会丢失，而是变成透明组件图片。
6. 组件图片在画板中可正常移动和缩放。
7. 全局背景不会错误导入为白底大图。
8. 对 `final_edited.svg` 这类样例，能得到“文本 + 连接箭头 + 组件图”的合理混合结果。

## 14. 已知风险

### 高风险

- 复杂 `path` 误判为箭头
- 文字基线偏移
- 旋转与嵌套变换处理不完整
- 组件边界划分不稳定

### 中风险

- 不同 SVG 导出器结构差异大
- 局部白底与全局背景的误判
- 连接箭头自动绑定到节点的准确率

### 低风险

- SVG 基础解析
- 文本提取
- 组件图片落板

## 16. 本方案结论

本方案的明确结论如下：

- Drawnix 适合做 **SVG 半结构化导入**，不适合当前阶段直接做通用 SVG 编辑内核。
- 最优首版路径是：
  - 文字转可编辑
  - 连接箭头按需转可编辑
  - 原形态箭头可直接作为组件导入
  - 复杂组件转透明图片
- 这一路线既满足核心需求，也能与当前 Drawnix 架构自然衔接。

后续开发默认以本文件作为 SVG 导入功能的执行基线。
